标准工作流程

当你在你的项目上工作时，你自然会创建sprites ，编辑scripts ，删除timelines ，以及其他任何随着你的项目发展而需要的东西。这些都是你可能想要跟踪的行动，如果出了问题，你可以找回来。这是使用Source Control 的主要原因--这样任何事故都可以被还原--我们将迅速带你从头开始学习基本的工作流程，以便你了解这一点。

从头开始，创建一个新的GML 项目，然后通过主选项激活source control 。这样我们就可以进入顶部的Source Control 菜单，所以我们马上去选择创建项目资源库。如果你看一下room 资源，你可以看到默认的room 已经获得了一个红色的感叹号 ，这意味着它已经从最后的状态被修改了 。如果我们在项目中还有其他资源，你会看到它们都有同样的图标，如下图所示。

我们现在可以选择提交空白项目，或者把它变成一个基本状态。现在，我们将进行第一次提交，所以点击源代码控制>提交更改。现在我们会看到一个新的窗口，它被分成三个区域。

  • 阶段性变化。这代表将被提交到源代码的变化。这可能是空白的，但它可能会显示一个主选项的变化，这很好。
  • 未归档的修改。这显示了已经改变的文件，但我们还没有告诉Git 我们要提交它们 - 它们是 "待定 "的改变。
  • 提交信息。这是我们可以添加的说明，以解释什么是变化。

现在，点击Stage All按钮，因为我们想提交所有的修改，然后在Commit Message中输入类似 "First Commit!"的字样，并点击Commit。现在我们所有的资源都应该有一个绿色的勾 ，意味着没有发现任何修改。

值得注意的是，当你在这个窗口中工作时，各种阶段性 和非阶段性的 信息将以一个符号为前缀。所使用的符号将根据操作和文件的不同而不同，以下是使用的符号。

  • [...] - 阶段性变化（以前是[A]）。
  • [+] - 新添加
  • [->] - 重新命名/移动
  • [-] - 已删除
  • [M] - 非阶段性变化
  • [?] - 未被追踪的文件
  • [!] - 缺少文件

我们的工作进程现在可以开始了，我们可以开始建立我们的项目，并保存和提交修改，就像我们在上面几段所做的那样。

在工作了一段时间后，你可能想检查并查看你在任何时候都提交了什么，因此，你要打开提交历史。要打开历史记录窗口，你只需进入源代码控制>查看历史记录，这将打开另一个有三个窗格的窗口。

上面的窗格描述了提交的内容，中间的窗格描述了给所选提交的注释，下面的窗格描述了在这个提交中被修改的文件。注意，如果你已经设置了差分工具，那么双击这个窗口中的任何文件都会打开差分工具，你就可以看到文件之间的变化。
